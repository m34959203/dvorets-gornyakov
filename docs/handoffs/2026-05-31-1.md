## Что сделано

### 1. URL-агностичность (главный фикс) — og:image/canonical больше не ломаются при смене туннеля
**Проблема Cowork:** og:image указывал на мёртвый `counting-zero-place-track.trycloudflare.com` (NXDOMAIN) → шеринг в соцсети ломался.
**Корень:** `metadataBase`/canonical/share-URL строились из вшитого в билд `NEXT_PUBLIC_APP_URL`, тогда как sitemap/robots читали `site_base_url` из БД. При каждой смерти quick-туннеля метаданные уезжали на старый домен до ребилда. (Та самая ловушка из правила «Next.js APP_URL baked».)
**Решение:**
- Новый общий хелпер `src/lib/site-url.ts` → `getSiteBaseUrl()` читает `site_base_url` из БД (env/дефолт — фолбэк).
- Переведены ВСЕ 17 мест: корневой `layout.tsx` (`metadataBase`), generateMetadata всех публичных страниц (canonical+alternates), share-URL и JSON-LD `url` на деталках событий/новостей.
- Статическим страницам (`about`, `rules`, `resources`) добавлен `export const dynamic = "force-dynamic"`, иначе значение из БД забэкалось бы на сборке.
- **Доказано:** подмена `site_base_url` в БД мгновенно меняет canonical/og **без ребилда** (проверено на тестовом URL, затем возвращено).

**Остаточный нюанс (осознанно):** внутренние base в `src/lib/jsonld.ts` (org `@id`, breadcrumb URL в `DgPageHero`) всё ещё читают `NEXT_PUBLIC_APP_URL` — это синхронный презентационный компонент на всех страницах, async-DB-вызов туда добавлять не стал (был бы DB-хит на каждый рендер). На каждый деплой они корректны (`.env` синхронизирован с живым URL); при смене туннеля между деплоями слегка устаревает только JSON-LD `@id`/breadcrumb (минорный SEO-сигнал), а пользовательские canonical/og/share — уже динамические.

### 2. Тайтл деталки кружка без бренда — ПОЧИНЕНО
**Было:** `<title>Вокальная студия</title>` (на других — `… · Дворец горняков · Сатпаев`).
**Корень:** `clubs/layout.tsx` задаёт строковый `title`, что в Next.js рвёт наследование `title.template` из `(public)/layout.tsx` к внуку `clubs/[id]`. У событий своего layout нет → они наследуют шаблон и работают.
**Стало:** в `clubs/[id]/page.tsx` бренд-суффикс собирается явно (`title.absolute`, локаль-зависимо).
**Проверка:** `Вокальная студия · Дворец горняков · Сатпаев` ✅

### 3. /contacts — теперь полноценная страница с 2GIS (было: редирект)
**Было:** `/contacts` = `permanentRedirect` на `/about#contacts`; карты 2GIS нигде (на главной — Google Maps).
**Стало:** реальная страница `/ru|kk/contacts` (200) — DgPageHero + блок контактов (адрес/2 телефона/email/часы/соцсети) + форма обратной связи + карта-панель 2GIS (deep-link `2gis.kz/satpaev/search/…` по адресу пр. Сатпаева 106). CSS-стили карты добавлены в `globals.css` (`.contact-map-card`).
**Почему не live-iframe:** встроенный интерактивный виджет 2GIS требует ключа MapGL / зарегистрированного виджета под аккаунтом 2GIS учреждения. `2gis.kz` отдаёт 403 на фрейминг, legacy `widgets.2gis.com` без ключа = «Something went wrong». Кривой пин клиенту хуже корректного deep-link. **TODO для заказчика:** получить 2GIS-виджет/ключ → заменить панель на iframe (структура уже под это; в коде помечено комментарием).

### 4. TTS чат-бота — осознанно НЕ делаем браузерный (решение: STT-only)
Cowork рекомендовал включить `speechSynthesis`. **Отклонено** — браузерный TTS запрещён правилом проекта (RU=Gemini→GCloud, KK=ElevenLabs; иначе нестабильно/503). Озвучка ответа из чек-листа готовности убрана. Голосовой ВВОД (STT-микрофон) остаётся. Если озвучка понадобится — делать серверным Gemini-TTS отдельной задачей.

## Ложные / неподтверждённые находки Cowork

- **`/api/analytics/event` → 503**: НЕ воспроизводится. Маршрут отдаёт 204 (с cookie `dg_sid`) и 400 `no_session` (без неё); пути к 503 в коде нет вообще (даже внутренняя ошибка → 204). Вероятно транзиент в момент перезапуска туннеля/контейнера.
- **`/ai` → 404**: ложная находка. Ни в навигации (Header/MobileNav/Footer), ни где-либо ссылки на `/ai` НЕТ, страницы тоже нет — Cowork потыкал угаданный роут. Убирать нечего.
